# Protect Go 사용자 가이드 · 자동 업데이트 에이전트 — 작업 계획 (v0.7)

> 작성: 이서영 · 검토 요청: 이근원 매니저 · 기준일: 2026.06.05
> 변경(v0.7): **KB 구축 고도화 방향 추가** — 여러 소스를 조합해 KB 엔트리를 만들되 소스별 담당 필드를 분리(화면설계서는 정책 텍스트만, 화면 세부는 프론트엔드 코드 참조). 이를 반영해 KB 엔트리 스키마를 필드별 출처(provenance) 구조로 고도화.

## 목적
**지식 베이스(KB) 콘텐츠를 구조화하고, 그 위에서 Protect Go 최신버전 사용자 가이드(웹페이지)를 만든다.** 가이드를 만드는 작업이 곧 KB 콘텐츠를 구조화하는 작업이다. 자동 갱신은 같은 스키마 위에서 동작하도록 **설계만 해두고(스텁: 인터페이스만 정의하고 구현은 비워둔 상태)**, 구현은 후속으로 둔다.

## 범위 (2주 안에 끝내는 것)
- **가이드 페이지 + KB 콘텐츠를 하나의 묶음으로 진행.** 둘을 분리하지 않는다(가이드 구조화가 곧 KB 콘텐츠 구축).
- **자동 업데이트 에이전트는 스텁**: KB 엔트리 스키마(=인터페이스 계약)와 흐름 설계도까지만. 가이드는 수동 구축한 실제 KB 콘텐츠 위에서 동작하고, 에이전트는 같은 스키마에 나중에 끼워 넣는다(contract-first: 인터페이스를 먼저 확정하고 구현은 나중에 붙이는 방식).
- **도메인 범위 축소**: 9개 카테고리를 다 하지 않고 **핵심 1~2개 도메인만 깊게**(제안: 프로젝트 생성 + 탐지 시나리오·노드). 나머지는 같은 템플릿으로 "확장 가능함"만 시연.
- **화면 처리·소스**: 이미지 대신 HTML/CSS(또는 실제 배포 프론트엔드)로 직접 렌더링한다. **2주 범위에선 정책 텍스트는 화면설계서 PDF에서, 화면 세부(레이아웃·UI)는 프론트엔드 코드에서 참조**하며, **Figma 직접 연동은 후속(범위 외)** 이다. 언어는 한국어 우선(영어는 여유 시).

## 전제
- 이미 구조화된 `protect-go-knowledge`(60여 개 PDF에서 정리한 10개 도메인, 2026.04 기준)를 KB 콘텐츠 출발점으로 재활용·확장.
- 인풋 산출물 역할은 정기회의 확정안 기준(아래 매핑 표).

## 진행 방법론
1. **사전조사(딥리서치)** — 실리콘밸리·빅테크의 KB 구축·운영 방식 조사로 설계 기준 확보: RAG, 사내 위키, **doc-as-code**, 지식 그래프, 에이전트용 KB / 엔트리·메타데이터·버저닝·출처 추적 / 문서 변경 시 자동 반영(증분 인덱싱·재생성). 콘텐츠 모델로는 **Diátaxis**(튜토리얼·하우투·레퍼런스·설명)를 채택한다.
2. **PRD 플로우(선택)** — 필요 시 PRD 플로우(`prd-builder`, Notion)로 진행. 2주 범위에선 가이드·KB에 집중하고, 본격 PRD 작성과 **AI 에이전트 포함 모드(Phase 3-A)** 는 후속에서 다룬다.

## 단계별 계획 (2주 범위 + 후속)
| 구분 | 내용 | 산출물 |
|---|---|---|
| **1주차 — 정의·범위 설정 + 콘텐츠 구조화** | IA·카테고리 확정(Diátaxis × 도메인), KB 엔트리 스키마 확정, **소스→필드 매핑(조합 규칙) 설계**, 페르소나·태스크를 정의한 뒤 지표 설계, 대상 1~2개 도메인을 선정해 KB 콘텐츠 구조화 착수 | IA + 스키마 + 조합 규칙 + 지표 설계 / KB 콘텐츠(착수) |
| **2주차 — 1차 렌더링 + 스텁 + 지표** | 가이드 웹페이지 1차 렌더링(HTML/CSS, 한국어)과 다듬기 / 자동 업데이트 에이전트 스키마·흐름 스텁 / 지표 측정 / 포트폴리오 정리 | 가이드(한국어) + 에이전트 설계 + 지표 |
| **범위 외 (후속)** | 다중 소스 자동 조합 구현, 에이전트 구현(4모듈 전체), 전 도메인 확장, 영어판, 팀 업무 파이프라인 실연동(Figma 연동 포함), PRD 플로우 본격 진행 | — |

## KB 카테고리 (초안 — 기존 10개 도메인 기반)
사용자 가이드 관점으로 재배열한 카테고리:
1. 시작하기 / 제품·용어 2. 계정·권한 3. 프로젝트 관리 4. 탐지 시나리오·노드 구성
5. 대시보드·통계 6. 데이터·모델 관리 7. 모바일 앱 8. 어드민·공통 9. 정책·문제해결(FAQ)
- **2주 범위는 3·4번(프로젝트 + 노드) 중심**, 각 카테고리 안에서 Diátaxis 4유형으로 페이지 구성.
- **확장 가능**: 신규 도메인 발생 시 카테고리 추가, 카테고리 내 하위 토픽 증분 추가.

## 인풋 데이터(산출물)별 역할 & KB 매핑
정기회의 확정안 기준. 각 산출물이 무엇을 담고/담지 않으며, 가이드 KB의 어느 부분을 채우는지 정리:

| 산출물 | 역할 | 담지 않는 것 | KB·가이드 활용 |
|---|---|---|---|
| PRD | 기능의 존재 이유, 범위, 성공 기준(high-level) | 컴포넌트 동작 상세, 계산 수식 | 가이드 "개요·목적·범위" + 기능 단위 식별 |
| 화면 프로토타입 | 화면 레이아웃 + 컴포넌트 동작 정책(상태 전환·수식·엣지케이스) | 기능 배경·문제 정의, 개발 단위 | **가이드 본문 1차 소스** — 동작·단계·상태/예외 설명 |
| Figma | 시각 완성도, variant, 인터랙션 상태 | 정책·기능 정의, 컴포넌트 동작 상세 | 화면 시각 자료(HTML/CSS 렌더링 기준), variant·상태 표시 |
| 티켓 | 구현 단위 분해 + 동작 요구사항 AC | PRD 내용 복사, 구현 방식·API 스펙 | 동작 요구사항(AC)으로 가이드 정확성·완전성 보강(누락 점검) |

> ※ 위 표는 **(향후) 자동 업데이트 에이전트가 받을 인풋 산출물의 역할 정의**다. 2주 범위의 실제 입력과는 구분한다.

- **현재 활용(2주 범위)**: GS 인증(Good Software 인증) 당시 사용설명서, 최신 화면설계서 PDF, `protect-go-knowledge` 10개 도메인, 실제 프론트엔드 코드.
- **향후 확장**: 신규 기능 PRD·티켓, Figma·프로토타입(직접 연동), 릴리즈 노트, 문의/FAQ, 멀티미디어, 추가 언어.

## 팀 업무 프로세스(확정) & 가이드 자동 업데이트 연계
```
PRD 작성 → Notion 저장
  → 화면 프로토타입 자동 생성 (protect-go-design 스킬)
  → PM 정책 검증 (PRD 반영·요구사항 충족 확인)
  → 디자이너: 초안 시각 검토 + Figma 상세 작업
  → Figma 결과물 프로토타입 병합 (MCP, 1차 PM 수동)
  → 정책 툴팁 삽입 (컴포넌트 상태 전환·수식·예외)
  → 티켓 자동 생성 (Notion Epic/Story/Task)
  → 개발자 전달
```
- **가이드 자동 업데이트 연계**: 위 산출물이 생성·갱신되는 각 시점이 (향후) 가이드 자동 업데이트 에이전트의 인풋 트리거가 된다. 특히 정책 툴팁까지 반영된 **화면 프로토타입이 가이드 본문의 1차 소스**다. (2주 범위에선 트리거 없이 수동으로 KB 콘텐츠를 채운다.)
- **주의(현황)**: ① Figma 결과물의 프로토타입 병합 자동화는 추가 테스트가 필요해 현재 PM이 1차 수동 수행. ② 인터랙션 정의는 엄밀히 디자이너 영역이나 점진적 정리 방향 유지.

## KB 구축 고도화 — 소스 조합 전략 (방향성)
하나의 KB 엔트리는 단일 문서를 그대로 옮긴 것이 아니라, **여러 소스에서 필요한 부분만 뽑아 조합**해 만든다. 소스마다 담당(권위) 필드를 분리해 중복과 드리프트(문서-제품 불일치)를 줄이는 것이 핵심이다.

**필드 → 권위 소스**
- 기능 목적·범위(개요): **PRD**
- 정책 텍스트(상태 전환·수식·예외): **화면설계서(PDF)에서 정책 부분만 추출**(전체 내용이 아니라 정책 위주) + 화면 프로토타입
- 화면 세부(레이아웃·컴포넌트·실제 UI 텍스트·상태): **프론트엔드 코드 참조**(이미지·문서 캡처 대신 실제 코드를 단일 출처로)
- 동작 요구사항(AC): **티켓**
- 시각 자료·variant: **Figma**(후속)

**왜 고도화인가**
- 화면 세부를 코드에서 끌어오면 화면이 바뀔 때 가이드가 자연히 따라가 드리프트가 최소화된다. 문서(PDF)는 잘 변하지 않는 **정책**만 담당한다.
- 필드별로 출처·버전을 기록(provenance)하면, 특정 소스가 갱신됐을 때 **그 필드만 다시 생성**하면 된다. 증분 업데이트의 토대가 된다.

**범위 구분**
- **2주(설계)**: 소스→필드 매핑(조합 규칙)을 정의하고, 선정 도메인에 수동 적용해 한두 엔트리 예시를 만든다.
- **후속(구현)**: 다중 소스에서 필드별로 자동 추출·조합하는 에이전트.

## KB 엔트리 스키마 (고도화 — 필드별 출처 반영)
가이드·KB·(향후)에이전트가 공유하는 **인터페이스 계약**. 한 엔트리가 여러 소스의 조합임을 `provenance`로 드러낸다.
```
{
  "id", "category", "diataxis_type",   // 튜토리얼|하우투|레퍼런스|설명
  "title", "description",              // ← PRD (기능 목적·범위)
  "policy": "",                        // 상태 전환·수식·예외 ← 화면설계서(PDF) 정책 텍스트 + 화면 프로토타입
  "screen_ref": "",                    // 화면 세부(레이아웃·컴포넌트·UI 텍스트) ← 프론트엔드 코드 참조
                                       //   캡처 대상 엔트리는 매핑형 { notes, page, capture, hotspots }
                                       //   (schema/kb-entry.md "screen_ref 구조형" — capture_screens.py 입력)
  "acceptance": "",                    // 동작 요구사항(AC) ← 티켓
  "visual": "",                        // 시각 자료·variant ← Figma (후속)
  "provenance": {                      // 필드별 출처·버전 (증분 업데이트 기준)
    "policy":     { "source": "화면설계서(PDF)", "version": "" },
    "screen_ref": { "source": "frontend_code",  "ref": "", "version": "",
                    "captured_commit": "" }  // 화면 캡처 시점 커밋 — capture_screens.py가 캡처 성공 시 갱신
  },
  "language", "updated_at"
}
```

## 자동 업데이트 에이전트 로직 (스텁 — 설계만)
입력(PRD·화면설계서·프론트엔드 코드·화면 프로토타입·Figma·티켓)이 들어오면 다음 순서로 처리한다.
**①판별**(수정 vs 신규) → **②검색/매핑**(기존 KB에서 대상·필드 탐색) → **③구조화/생성**(필드별 권위 소스에서 추출·조합해 스키마에 맞춰 생성 또는 해당 필드만 수정) → **④반영**(DB 저장 후 웹페이지 반영). 분기 처리는 LangGraph 등으로 구성 가능. (2주 범위에선 흐름 설계도와 인터페이스 스키마까지만.)

## 정량 지표 설계 방향
- **대상·목적(페르소나)** 이 정해지면 지표는 논리적으로 도출.
- (A) 지식 베이스 관점: 정보 검색 비용 절감 / AX(AI 전환)에 따른 업무 효율·시간 단축.
- (B) 가이드 관점: 주 대상은 가이드 *만드는 사람* 이며, 기존 상세 설명서 작성 시간 대비 단축 효과를 수치화.

## 일정
- 기간 **2주**: 1주차 정의·범위 설정 + 콘텐츠 구조화, 2주차 1차 렌더링 + 스텁 + 지표. 회사 외 진행 가능.
