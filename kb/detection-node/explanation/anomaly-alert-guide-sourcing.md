# 이 가이드가 만들어지는 방식 — 출처와 반영 흐름

이 가이드의 모든 문장에는 출처가 있다. 어떤 출처에서 어떤 내용이 오고, 출처가 바뀌면 가이드가 어떻게 따라가는지 설명한다. [이상 상황 알림 확인하기](../how-to/handle-anomaly-situation.html)가 이 방식으로 만들어진 첫 사례다.

## 출처별 역할

가이드 한 편은 단일 문서가 아니라 여러 출처의 조합이다. 필드마다 권위 있는 출처가 다르다.

| 가이드 내용 | 출처 | 권위 |
|---|---|---|
| 제목·기능 요약 | PRD (기획 문서) | 확정 |
| 정책 (상태 전환·유효성·예외) | 화면설계서 PDF의 정책 추출분 | 확정 |
| 화면 세부 (버튼·문구·구성) | 프론트엔드 코드 | 확정 |
| 동작 요구사항 | 티켓(수용 기준) | 확정 |
| 위 출처가 아직 없는 신규 기능 | **배포된 프로토타입 (라이브 URL)** | **draft (임시)** |

## draft 출처 — 배포 프로토타입

이상 상황 알림처럼 화면설계서·프론트엔드 코드가 아직 없는 신규 기능은, 배포된 프로토타입(예: `protect-go-react.vercel.app`)을 **임시(draft) 출처**로 삼아 가이드를 먼저 제공한다.

- draft 출처로 작성된 내용에는 "배포 프로토타입 기준, 확정 사양 아님" 안내가 붙는다.
- **draft 출처는 텍스트에만 쓴다.** 가이드에 들어가는 화면 이미지와 번호 핀은 항상 실제
  솔루션 화면에서 캡처한다 — 프로토타입 화면을 이미지로 쓰면 사용자가 실제 화면과 다른
  안내를 받기 때문이다.
- 권위 등급은 항상 **확정 출처가 draft 출처보다 우선**한다. 화면설계서·코드가 나오면 그 기준으로 내용을 교체(승격)하고, 두 출처가 충돌하면 확정 출처가 이긴다.
- 프로토타입과 실제 제품의 동작이 다를 수 있다 — draft 기반 가이드는 "지금 설계가 이렇다"를 알려주는 것이지, 출시 사양을 보증하지 않는다.

## 반영 흐름 — 출처가 바뀌면 가이드는 어떻게 따라가나

1. **페치**: 점검 도구가 프로토타입 배포 URL(항상 최신 배포)을 다시 읽어 내용 지문(content hash)을 만든다.
2. **변경 감지**: 직전 스냅샷과 비교해 내용이 바뀐 부분을 찾아낸다. 바뀐 부분이 없으면 여기서 끝난다.
3. **사람 검토**: 변경이 감지되면 담당자가 바뀐 내용을 직접 보고, 가이드 본문을 수정할지 판단한다. 본문을 자동으로 다시 쓰지 않는다.
4. **가이드 갱신**: 수정이 필요하면 사람이 해당 부분만 고치고, 출처 기록(어느 배포의 어느 시점 내용인지)을 함께 갱신한다.
5. **빌드·게시**: 갱신된 내용으로 가이드 사이트를 다시 만든다. 화면 캡처도 같은 방식으로 — 코드나 배포가 바뀐 화면만 — 다시 찍는다.

프론트엔드 코드 출처도 같은 흐름을 따른다. 점검 도구가 코드 저장소의 최신 버전을 받아온(pull) 뒤, 가이드가 참조한 파일의 변경 여부와 화면 캡처의 최신 여부를 함께 점검한다. 코드가 바뀐 화면만 다시 캡처해 가이드를 갱신한다.

요약하면: **자동으로 감지하고, 사람이 판단해 고치고, 바뀐 부분만 갱신한다.** 그래서 가이드가 제품보다 오래된 채 방치되지 않으면서도, 검토 없이 잘못된 내용이 실리지 않는다.

## 예고된 개편 추적 — 이상 상황 레이어 (draft 추적 사례)

알림 센터의 다음 단계로, 이상 상황을 문제 단위로 확인·조치·감사까지 한 흐름으로 처리하는
**"이상 상황 레이어" 개편**이 프로토타입(`protect-go-react.vercel.app`)으로 공개되어 있다.
이 가이드는 해당 프로토타입을 draft 출처로 등록해 추적 중이다.

프로토타입이 예고하는 개편의 핵심은 다음과 같다 (draft — 출시 사양 아님).

- 이상 상황 하나가 여섯 단계를 거쳐 지식으로 축적된다: AI가 탐지 이벤트를 클러스터링하고,
  이상 상황으로 제안하고, 매니저가 확인·수락하고, 현장 조치를 기록하고, 에디터가 사후
  감사하고, 지식 베이스에 축적된다.
- 상태는 6단계로 관리된다: 확인 대기 / 확인 중 / 조치 완료 / 처리 지연(7일 초과) /
  이의제기 중 / 이상 없음(오탐 종료).
- 축적된 조치 이력은 다음 유사 상황에서 "유사 사례 요약"으로 자동 추천된다.

프로토타입 내용이 바뀌면 변경 감지 도구가 알려주고, 사람이 검토해 이 절을 갱신한다.
개편이 실제 제품에 반영되면 확정 출처(화면설계서·코드) 기준으로 승격되어
[이상 상황 알림 확인하기](../how-to/handle-anomaly-situation.html) 가이드 본문에 합쳐진다.
이 절에는 화면 이미지를 싣지 않는다 — 가이드 이미지는 실제 솔루션 화면만 쓰기 때문이다.

## 이 방식의 의도

- **어긋남(드리프트) 방지**: 가이드와 제품이 어긋나는 것은 대부분 "출처가 바뀐 줄 몰라서"다. 출처별 버전 기록과 변경 감지로 어긋남을 빨리 찾는다.
- **신규 기능 공백 제거**: 확정 문서를 기다리는 동안에도 draft 출처로 가이드를 먼저 제공한다. 사용자는 빈 페이지 대신 "현재 설계 기준" 안내를 받는다.
- **증분 갱신**: 출처 하나가 바뀌어도 가이드 전체를 다시 쓰지 않는다. 영향을 받은 필드만 고친다.
