# 01. 문제 정의

## 메인 문제
가이드/설명서를 만드는 사람이 화면·정책이 바뀔 때마다 문서를 수작업으로 갱신해야 해서 최신성이 떨어지고(드리프트), 작성 비용이 크다.

- **분류 이유**: 계획서 v0.7의 핵심 동인이 "가이드를 만드는 작업 = KB 콘텐츠를 구조화하는 작업"이며, 이번 작업의 설계 무게중심이 "여러 소스를 필드별 권위로 분리해 KB 엔트리를 조합하고, 변경분만 증분 갱신"하는 데 있다. 즉 이 프로젝트가 직접 풀려는 문제는 "사람이 만드는 문서가 제품을 따라가지 못하는 문제"다.

## 부차 문제
Protect Go 사용자가 기능 사용 중 막히면 참고할 최신 가이드가 없어 시행착오를 겪거나 직접 문의해야 한다.

- **분류 이유**: 메인 문제(드리프트 없는 KB 구조)가 해결되면 자연히 따라오는 결과이자 가이드의 최종 수혜 지점이다. 이번 작업에서 직접 측정·검증하지는 않지만, 콘텐츠 모델(Diátaxis)·페르소나 설계의 근거가 된다.

## 범위 외
여러 소스(PRD·화면설계서·Figma·코드·티켓)에 흩어진 정보를 매번 새로 찾아 조합해야 해서, 신규 합류자나 타 팀이 제품 동작을 파악하는 데 시간이 오래 걸린다.

- **분류 이유**: 조직 전체의 지식 검색 비용 절감은 더 큰 비전이지만, 2주 범위(가이드 페이지 + KB 콘텐츠 묶음, 1~2 도메인)를 넘어선다. 자동 업데이트 에이전트(스텁 설계까지만)와 다중 소스 자동 조합이 본격 구현되는 후속 단계의 영역이다.
