# Protect Go AI 사용자 가이드 — Full PRD

> Status: Discovery 완료 | 최종 수정: 2026-06-10
> 컨텍스트: Protect GO AI (PG) | 모드: One-Page | 이해관계자: 이서영(작성), 기획팀(검토)

## 문제 정의

- **메인 문제**: 가이드/설명서를 만드는 사람이 화면·정책이 바뀔 때마다 문서를 수작업으로 갱신해야 해서 최신성이 떨어지고(드리프트), 작성 비용이 크다.
  - 분류 이유: 계획서 v0.7의 핵심 동인이 "가이드를 만드는 작업 = KB 콘텐츠를 구조화하는 작업"이며, 설계 무게중심이 "여러 소스를 필드별 권위로 분리해 KB 엔트리를 조합하고, 변경분만 증분 갱신"하는 데 있다.
- **부차 문제**: Protect Go 사용자가 기능 사용 중 막히면 참고할 최신 가이드가 없어 시행착오를 겪거나 직접 문의해야 한다. 메인 문제가 해결되면 자연히 따라오는 결과이자 가이드의 최종 수혜 지점이다.
- **범위 외**: 여러 소스에 흩어진 정보를 매번 새로 찾아 조합해야 해서 신규 합류자·타 팀이 제품 동작을 파악하는 데 시간이 오래 걸린다. 조직 전체 지식 검색 비용 절감은 더 큰 비전이나, 2주 범위와 자동 업데이트 에이전트(스텁)의 후속 영역이다.

## 핵심 가치 가설

기존 상세 설명서 작성 방식 대비 가이드 제작·갱신 시간을 단축한다.

KB 엔트리를 "여러 소스의 조합 + 필드별 출처(provenance)"로 구조화하면, 화면·정책이 바뀌었을 때 영향받는 필드만 다시 만들면 된다. 갱신 단위를 줄여 제작·갱신 시간 자체를 단축하는 구조적 가설이며, 계획서의 지표 설계 방향 "가이드 관점: 주 대상은 만드는 사람, 작성 시간 단축 수치화"와 정렬한다.

| 페르소나 | 분류 | Job-to-be-Done |
|---|---|---|
| KB(Knowledge Base)/가이드 구축 담당자 (이서영 등) | 메인 | 화면·정책이 바뀔 때 최소 노력으로 가이드를 최신 상태로 유지하고 싶다 |
| Protect Go 일반 사용자 | 서브 | 기능 사용 중 막히는 지점에서 바로 답을 찾고 싶다 |
| 어드민·매니저 | 서브 | 자기 권한 범위에 맞는 설정·정책 정보를 빠르게 확인하고 싶다 |
| (향후) 자동 업데이트 에이전트 | 범위 외 | 변경된 소스를 감지해 같은 스키마로 KB를 갱신 — 스키마·흐름 스텁까지만 |

## 솔루션 범위

### 채택 솔루션 방향
S1(KB 엔트리 스키마 적용 + 실제 엔트리 1~2개 제작·검증)을 토대로, S2(Diátaxis 콘텐츠 템플릿 — Diátaxis는 튜토리얼·하우투·레퍼런스·설명 4유형 콘텐츠 분류 모델)·S4(권위 소스 — 신규 문서 대신 기존 schema 표 단일화)·S5(소스 변경 영향 수동 점검 체크리스트)를 보조로 결합한다. S6(`kb/` to `guide/` 렌더링 빌드 스크립트)은 같은 V1 안에서 후반부로 순서를 고정한다.

### V1 범위 경계
1. **핵심**: 스키마 적용 + 실제 엔트리 제작·비교 검증 (Devil's Advocate 반론 — 검증 액션 없이는 가설이 추정에 머문다는 지적에 대한 직접 대응)
2. **보조**: 콘텐츠 템플릿·체크리스트(정책 복잡도별 갈래 구분), 권위 소스는 기존 표 단일화, 점검 체크리스트는 소스당 1~2문항으로 제한
3. **후반부 순서 고정**: 렌더링 빌드 스크립트 — 토대(엔트리 제작·검증) 완료 이후 착수

### 명시적 제외 (V2 / 범위 외)
- 자동 영향 탐지·자동 재생성(에이전트 본구현) — 페르소나 분류·CLAUDE.md 스텁 규칙과 직접 충돌(Hard Stop H1 대상이므로 발산 단계에서부터 제외)
- 정책 추출 파이프라인의 본격 도구화 — 2주 범위에서는 현재 스크립트(`extracted/build_policy_sources.py`) 수준 유지

## AI 에이전트 사양 (스텁)

자동 업데이트 에이전트는 인터페이스 계약과 흐름 설계까지만 다루며, 실제 동작 구현은 본 V1 범위 밖이다.

<details>
<summary>흐름 개요 및 V1과의 접점</summary>

판별(변경 소스 분류) → 검색·매핑(대상 엔트리·필드 특정) → 구조화·생성(필드별 권위 소스 조합) → 반영(KB 기록 및 빌드 트리거)의 단방향 4단계로 구성되며, 각 단계는 직전 단계의 산출만 입력으로 받는다.

공유 타입 KB 엔트리는 `schema/kb-entry.md` 프런트매터와 1대1 대응한다(식별자·카테고리·Diátaxis 유형·본문 필드·필드별 출처 정보·언어·갱신 시각).

V1의 F5(소스 변경 영향 수동 점검 체크리스트)는 흐름의 1·2단계(판별, 검색·매핑)를 사람이 어떻게 수행하는지 보여주는 선행 사례로, 향후 자동화 설계의 1차 입력이 될 수 있다.

비범위: 각 단계의 실제 구현, 오류·재시도·동시성 처리, 트리거 연결 — V1에서 착수 시 Hard Stop H1(기존 사양 충돌) 대상.
</details>

## 우선순위

**P0** — F1(스키마 적용), F2(엔트리 제작·검증), F6(가이드 출력): 가설 검증과 메인 페르소나 영향의 핵심
**P1** — F3(템플릿·체크리스트), F5(변경 영향 점검): 핵심 작업을 보강하는 병행 항목
**P2** — F4(권위 소스 안내), F7(렌더링 빌드 스크립트), F8(라이브 배포 소스 참조 + 동기화 트리거): 경량 보조 또는 순서 제약이 있는 후반부 항목

## QA 시나리오

<details>
<summary>전체 QA 시나리오 11개 보기</summary>

**메인 페르소나 (KB/가이드 구축 담당자)**
1. 신규 엔트리 작성 시 권위 소스 안내를 따라가면 출처 확인 단계에서 막히지 않는지
2. 템플릿으로 Diátaxis 분류를 판단할 때 기준이 모호해 재판단이 필요한 경우가 있는지
3. 복잡한 정책 화면에서 단일 템플릿의 누락 여부와 갈래 분리 템플릿의 효과 비교
4. 점검 체크리스트로 좁힌 영향 범위가 실제 영향 범위와 일치하는지
5. 좁혀진 필드만 갱신했을 때 기존 방식 대비 손댄 범위가 줄었는지 (갱신 단위 크기 KPI와 직결)
6. 신규 작성과 기존 방식의 소요 시간 차이가 기록 가능한 형태로 남는지

**서브 페르소나 (일반 사용자 / 어드민·매니저)**
7. 막힘 지점에서 가이드의 분류 구조를 따라 하우투·튜토리얼 항목을 찾을 수 있는지
8. 가이드의 설명·레퍼런스가 실제 화면 동작과 어긋나지 않는지
9. 권한 범위에 맞는 정보가 권한 밖 정보와 분리되어 노출되는지

**변경 반영 흐름**
10. 빌드 스크립트 실행 후 변경분이 가이드 화면에 정확히 반영되는지

**경계 시나리오**
11. 소스 변경 시 자동 갱신이 동작하지 않고 사람이 점검 체크리스트로 직접 판단해야 한다는 경계가 명확히 인지되는지 (스텁 단계의 정상 동작 검증)

</details>

## 결정 로그

<details>
<summary>Silent Default 2건 및 자동 진행 요약 보기</summary>

**SD-001 · 비교 검증 대상 엔트리 선정 기준**
정책 복잡도가 높은 화면(탐지 시나리오·노드 1개 + 프로젝트 생성 1개)을 기본 선정. 여기에 더해 조치이력·알림 시스템 개편을 추가 선정한다 — 두 도메인은 프로토타입 코드가 존재하는 신규 기능이라 실제 동작 기반 테스트가 가능하고, screen_ref 필드 연동(프론트엔드 코드 참조) 검증에도 적합하다. 검토된 대안: 최단순 화면 우선, 무작위 선정.

**SD-002 · KPI 비교 기준선 측정 방법**
GS 인증 사용설명서 작성 경험이나 작성자 인터뷰 기반 추정치를 임시 기준선으로 설정. 근거: 별도 정량 기록 체계가 확인되지 않는 상황에서 가장 근접한 실측 경험 자료. 검토된 대안: 정성 비교만 수행, 기준선 측정을 후속으로 이연.

**자동 진행 요약**
