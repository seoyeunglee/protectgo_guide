# Protect Go AI 사용자 가이드 — One-Pager

> Status: Discovery 완료 | 최종 수정: 2026-06-08
> 본 페이지는 Macro Gate 1(Problem Space) 통과 시 작성된 Problem 섹션을 유지하면서, Macro Gate 1.5(Solution Space)·Gate 2(Discovery) 통과 시점에 솔루션 범위·KPI 섹션을 추가한 전체 1-Pager입니다. 세부 항목은 하위 페이지 "Full PRD"에서 확인할 수 있습니다.

## 문제 정의

- **메인 문제**: 가이드/설명서를 만드는 사람이 화면·정책이 바뀔 때마다 문서를 수작업으로 갱신해야 해서 최신성이 떨어지고(드리프트), 작성 비용이 크다.
  - 분류 이유: 계획서 v0.7의 핵심 동인이 "가이드를 만드는 작업 = KB 콘텐츠를 구조화하는 작업"이며, 설계 무게중심이 "여러 소스를 필드별 권위로 분리해 KB 엔트리를 조합하고, 변경분만 증분 갱신"하는 데 있다.
- **부차 문제**: Protect Go 사용자가 기능 사용 중 막히면 참고할 최신 가이드가 없어 시행착오를 겪거나 직접 문의해야 한다.
- **범위 외**: 여러 소스에 흩어진 정보를 매번 새로 찾아 조합해야 해서 신규 합류자·타 팀이 제품 동작을 파악하는 데 시간이 오래 걸린다 — 자동 업데이트 에이전트(스텁)와 함께 후속 영역.

## 핵심 가치 가설

기존 상세 설명서 작성 방식 대비 가이드 제작·갱신 시간을 단축한다. KB 엔트리를 "여러 소스의 조합 + 필드별 출처(provenance)"로 구조화하면, 화면·정책이 바뀌었을 때 영향받는 필드만 다시 만들면 되므로 갱신 단위가 줄어든다.

## 솔루션 범위 (V1)

S1(KB 엔트리 스키마 적용 + 실제 엔트리 1~2개 제작·검증)을 토대로, S2(Diátaxis 콘텐츠 템플릿)·S4(권위 소스는 신규 문서 대신 기존 schema 표 단일화)·S5(소스 변경 영향 수동 점검 체크리스트)를 보조로 결합한다. S6(`kb/` to `guide/` 렌더링 빌드 스크립트)은 토대·출력 형태 확정 이후 후반부로 순서를 고정한다.

명시적으로 제외한 항목: 자동 영향 탐지·자동 재생성(에이전트 본구현 — 범위 외 분류·CLAUDE.md 스텁 규칙과 직접 충돌), 정책 추출 파이프라인의 본격 도구화(현재 스크립트 수준 유지).
