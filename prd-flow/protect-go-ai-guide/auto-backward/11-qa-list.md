# 11. QA 시나리오 목록

> 기준: 메인 페르소나(KB/가이드 구축 담당자) JTBD + 서브 페르소나(일반 사용자, 어드민·매니저) 시나리오
> 입력: `gate1/03-personas.md`, `auto-backward/07-features.md`

## 메인 페르소나 — KB/가이드 구축 담당자

1. **신규 엔트리 작성**: 스키마 구조(F1)에 맞춰 신규 엔트리를 작성할 때, 필드별 권위 소스 안내(F4)를 따라가면 출처 확인 단계에서 막히지 않는지 확인한다.
2. **Diátaxis 분류 판단**: 템플릿(F3)을 사용해 작성 중인 화면이 튜토리얼·하우투·레퍼런스·설명 중 어디에 해당하는지 판단할 때, 분류 기준이 모호해 다시 처음부터 판단해야 하는 경우가 있는지 확인한다.
3. **복잡한 정책 화면 작성**: 노드·조건 분기가 있는 화면(예: 탐지 시나리오·노드)을 작성할 때, 단일 템플릿으로는 누락되는 항목이 실제로 발생하는지, 갈래를 나눈 템플릿이 그 문제를 해소하는지 비교한다.
4. **변경 영향 점검**: 소스 한 곳이 바뀌었을 때 점검 체크리스트(F5)를 사용해 영향받는 엔트리·필드를 좁힌 결과가, 실제로 영향받은 범위와 일치하는지 확인한다.
5. **갱신 작업 수행**: 점검으로 좁힌 필드만 다시 작성했을 때, 문서 전체를 다시 검토했던 기존 방식과 비교해 실제로 손댄 범위가 줄었는지 확인한다 (KPI "갱신 단위 크기"와 직접 연결).
6. **제작 시간 비교**: F2에서 제작한 엔트리의 소요 시간을, 같은 화면을 기존 방식으로 작성했을 때의 시간과 비교했을 때 차이가 기록 가능한 형태로 남는지 확인한다.

## 서브 페르소나 — Protect Go 일반 사용자

7. **막힘 지점에서 답 찾기**: 기능 사용 중 막혔을 때, 가이드(F6)에서 해당 화면·상황에 맞는 하우투 또는 튜토리얼 항목을 분류 구조를 따라 찾아낼 수 있는지 확인한다.
8. **정책 이해**: 탐지 노드·시나리오처럼 조건이 복잡한 기능을 사용할 때, 가이드의 설명·레퍼런스 항목이 실제 화면 동작과 어긋나지 않는지 확인한다.

## 서브 페르소나 — 어드민·매니저

9. **권한 범위 정보 확인**: 자신의 권한 범위에 해당하는 설정·정책 정보를 가이드에서 찾을 때, 권한 밖 정보와 섞이지 않고 분리되어 노출되는지 확인한다.

## 변경 반영 흐름 (F7 관련)

10. **빌드 반영 확인**: KB 마크다운을 수정한 뒤 렌더링 빌드 스크립트(F7)를 실행했을 때, 변경분이 가이드 화면에 정확히 반영되는지, 반영되지 않은 항목이 있다면 어디서 누락됐는지 확인한다.

## 경계 시나리오 (제외 범위 확인용)

11. **자동 갱신 미동작 확인**: 소스가 변경돼도 KB·가이드가 자동으로 갱신되지 않으며, 사람이 점검 체크리스트(F5)를 통해 직접 판단·수정해야 한다는 경계가 실제 운영에서 명확히 인지되는지 확인한다 (자동 업데이트 에이전트는 스텁 단계이므로 자동 동작이 없는 것이 정상 동작임을 검증).
