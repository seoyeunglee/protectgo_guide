# 07. 기능 정의 (V1)

> 입력: `gate1.5/06-solution-scope.md` (채택: S1 토대 + S2·S4·S5 보조 + S6 후반부)
> 메인 페르소나: KB/가이드 구축 담당자 — JTBD "화면·정책이 바뀔 때 최소 노력으로 가이드를 최신 상태로 유지"

## F1 · KB 엔트리 스키마 적용 (선정 도메인)

`schema/kb-entry.md`의 필드별 provenance 구조를 프로젝트 생성, 탐지 시나리오·노드 두 도메인에 적용한다.

- 필드별 권위 소스 분리 적용: 제목·설명은 PRD, 정책(상태 전환·수식·예외)은 화면설계서 정책 추출본, 화면 세부는 프론트엔드 코드 참조, 인수 조건은 티켓
- 도메인별로 Diátaxis 4유형(튜토리얼·하우투·레퍼런스·설명) 분류 기준 확정
- `provenance.version` 필드를 증분 갱신 판단 기준으로 채워 넣음
- 적용 시 `research/kb-research.md`의 KB 설계 원칙 8개를 참고 기준으로 삼아, 스키마 구조가 실제 도메인 콘텐츠에서도 원칙과 어긋나지 않는지 점검하며 진행한다

## F2 · 실제 엔트리 제작 및 비교 검증

선정 도메인에서 엔트리 1~2개를 스키마 구조로 끝까지 제작하고, 기존 상세 설명서 작성 방식과 제작 과정을 비교한다.

- 제작 과정에서 필드별로 손댄 작업량을 기록 (가설 검증의 1차 데이터)
- 비교 결과를 핵심 가치 가설("제작·갱신 시간 단축")의 근거 자료로 남김
- Devil's Advocate moderate 반론(스키마만으로는 가설 입증 불가)에 대한 직접 대응 — V1에서 빠지면 안 되는 검증 액션

## F3 · Diátaxis × 도메인 콘텐츠 템플릿 및 작성 체크리스트

엔트리당 작성 시간을 표준화하기 위한 템플릿과 체크리스트를 제공한다.

- Diátaxis 4유형별 표준 구성 항목 정의
- 정책 복잡도가 낮은 화면용과 노드·조건 분기가 있는 화면용, 두 갈래로 템플릿을 구분 (Devil's Advocate 권고 반영 — 단일 템플릿은 복잡한 정책에서 누락을 유발할 수 있음)
- 작성 전 확인용 체크리스트(필드별 권위 소스 확인, Diátaxis 유형 적합성 확인)

## F4 · 권위 소스 안내 (신규 문서 생성 없음)

"어떤 필드를 어디서 가져오는지"를 확인하는 경로를 단일화한다.

- 별도 매핑 문서를 만들지 않고 `schema/kb-entry.md`의 필드 정의 및 권위 소스 표를 단일 기준으로 유지
- 작성 화면(템플릿, 체크리스트)에서 그 표를 가리키는 짧은 안내만 배치
- Devil's Advocate 반론(매핑 문서 자체도 드리프트 대상이 될 수 있음)을 구조적으로 차단하는 설계

## F5 · 소스 변경 영향 수동 점검 체크리스트

소스(PRD, 화면설계서, 코드, 티켓)가 바뀌었을 때 어떤 엔트리·필드가 영향받는지 사람이 빠르게 좁혀볼 수 있게 한다.

- 소스 1개당 1~2문항으로 짧게 구성 (체크리스트 길이가 길어지면 확인 비용이 늘어 가설과 반대로 작용한다는 Devil's Advocate 반론 반영)
- 상세 점검은 체크리스트 대신 `provenance.version` 비교로 대체
- 자동 탐지가 아닌 사람의 점검 보조 도구임을 명시 (자동 업데이트 에이전트와 명확히 분리)
- (F8) "라이브 배포 소스" 항목은 1~2문항: 배포 URL이 마지막 스냅샷 이후 바뀌었는가?
  (`content_hash`/`deployed_at` 비교) / draft로 채운 값이 확정 사양으로 승격됐는가?

## F6 · 가이드 웹페이지 출력

KB 엔트리를 실제 사용자가 보는 가이드 화면으로 노출한다.

- 이미지 대신 HTML/CSS(또는 실제 프론트엔드)로 렌더링
- Diátaxis 분류에 따른 탐색 구조 (튜토리얼·하우투·레퍼런스·설명)
- 일반 사용자, 어드민·매니저 서브 페르소나가 권한 범위에 맞는 정보를 찾는 경로 제공

## F7 · kb/ to guide/ 렌더링 빌드 스크립트 (후반부 — 순서 고정)

KB 마크다운 변경분이 가이드 화면에 반영되는지 확인하는 빌드 스크립트.

- F1·F2(토대)·F6(가이드 화면 형태) 확정 이후 착수 — 토대 없이 렌더링부터 만들면 재작업 위험이 있다는 Staff Engineer·Devil's Advocate 반론을 그대로 순서 제약으로 반영
- V1 범위 안에서는 "변경분 반영 확인" 수준까지만, 배포 자동화는 범위 외

## F8 · 라이브 배포 소스 참조 + 동기화 트리거 (후반부 — F7과 동일 그룹)

화면설계서 PDF·프론트엔드 코드 등 확정 출처가 아직 없는 신규 기능을, 배포된 프론트엔드
프로토타입(예: Vercel)을 임시(draft) 출처로 참조해 KB를 채울 수 있게 한다.

- `provenance.<field>.source: deployment`(`authority: draft`)로 인용 — 권위 등급은
  `schema 표(확정) > deployment(draft)`이며, 확정 출처가 생기면 그쪽으로 교체(승격)
- production alias URL(항상 최신 배포)을 페치해 `section_anchor` 단위로 변경을 감지하는
  점검·동기화 도구(`agent/scripts/fetch_deployment_source.py`)를 F5 점검 흐름에 추가
- 변경 감지 시 `provenance.<field>.deployment`의 `content_hash`/`fetched_at`/`deployed_at`
  메타데이터만 갱신(`sync --apply`) — 본문(`policy`/`screen_ref` 등) 자동 재생성은 하지 않음
- F7과 마찬가지로 토대(F1·F4) 확정 이후 착수하는 후반부 보조 기능

---
*F6은 F1·F2의 산출(구조화된 엔트리)을 사용자에게 보여주는 출력 통로이며, F7은 그 출력 과정을 자동화하는 후반부 보조 기능이다. 핵심 가치 가설 검증의 중심은 F1·F2에 있다.*
