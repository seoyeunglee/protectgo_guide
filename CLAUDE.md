# Protect Go 사용자 가이드 · KB 구축 — 프로젝트 메모리

> 이 파일의 규칙이 최우선이다. 충돌 시 CLAUDE.md를 따른다.

## 목적
지식 베이스(KB)를 구조화하고, 그 위에서 Protect Go 최신버전 사용자 가이드(웹페이지)를 만든다.
가이드를 만드는 작업 = KB 콘텐츠를 구조화하는 작업.

## 2주 범위 규칙 (반드시 준수)
- **묶음 진행**: 가이드 페이지 + KB 콘텐츠를 하나로. 둘을 분리하지 않는다.
- **도메인 축소**: 핵심 1~2개 도메인만 깊게 — 프로젝트 생성 + 탐지 시나리오·노드. 나머지는 템플릿으로 "확장 가능함"만 시연.
- **콘텐츠 모델**: Diátaxis(튜토리얼 / 하우투 / 레퍼런스 / 설명) × 도메인.
- **화면 처리**: 이미지 대신 HTML/CSS(또는 실제 프론트엔드)로 렌더링.
  - 정책 텍스트(상태 전환·수식·예외) → 화면설계서 PDF에서 **정책 부분만** 추출.
  - 화면 세부(레이아웃·컴포넌트·UI 텍스트) → **프론트엔드 코드 참조**.
  - **Figma 직접 연동은 하지 말 것(후속).**
- **언어**: 한국어 우선(영어는 여유 시).
- **에이전트는 스텁**: 자동 업데이트 에이전트는 스키마(인터페이스 계약) + 흐름 설계도까지만. **구현하지 말 것.**

## 범위 외 (하지 말 것 — 후속)
다중 소스 자동 조합 구현 / 에이전트 4모듈 구현 / 전 도메인 확장 / 영어판 / 팀 파이프라인 실연동(Figma 포함) / PRD 플로우 본격 구현.

## 파일 생성 원칙
- 작업에 필요한 폴더·파일은 위 구조에 맞춰 **먼저 묻지 말고 알아서 생성**한다.
- KB 엔트리 1개 = 파일 1개. 메타데이터·`provenance`는 YAML 프런트매터, 본문은 그 아래.
- 위 구조를 벗어나는 큰 변경(새 최상위 폴더, 스키마 구조 변경 등)만 진행 전에 묻는다.

## KB 엔트리 스키마 (단일 기준)
- 위치: `./schema/kb-entry.md`
- 한 엔트리는 여러 소스의 **조합**. 필드별 권위 소스:
  - `title`, `description` ← PRD
  - `policy`(상태 전환·수식·예외) ← 화면설계서 PDF 정책 텍스트 + 화면 프로토타입
  - `screen_ref`(화면 세부) ← 프론트엔드 코드 참조
  - `acceptance`(AC) ← 티켓
  - `visual` ← Figma (후속)
  - `provenance` ← 필드별 출처·버전(증분 업데이트 기준)
- 스키마 변경 시 계획서와 provenance 정의도 함께 갱신.

## 화면설계서 PDF 읽기 규칙 (작게, 정책만)
PDF를 통째로 컨텍스트에 읽지 말 것. 읽기 작업을 최대한 작게 만든다.
1. **텍스트만 추출**: 작은 스크립트로 PDF→텍스트 변환(`pymupdf`/`fitz` 또는 `pdftotext`). 이미지·레이아웃 무시. **페이지/화면 단위로** `extracted/<pdf명>/page-NN.txt` 저장. (스캔본일 때만 OCR.)
2. **정책만 필터**: 추출 텍스트에서 정책 신호(상태/전환/조건/예외/기본값/유효성/권한/제약/오류/필수 등)가 있는 섹션만 골라 `kb/_sources/policy/<screen>.md` 로 추린다. grep/정규식으로 1차 선별.
3. **추려진 것만 읽기**: 모델은 PDF가 아니라 추려진 정책 .md만 읽는다. 이번 도메인(프로젝트 생성·노드)에 해당하는 화면만.
4. **캐시 재사용**: 추출물은 `extracted/`에 남겨 한 번만 추출하고 재사용. (대용량·다수 PDF면 추출/필터를 서브에이전트(Task)에 위임해 메인 컨텍스트를 비운다.)
